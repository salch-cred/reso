Architecture Deep-Dive

How the compliance layer is built.

01

SEP-0057 Compliance Hook

A Soroban contract implementing Stellar's new compliance-hook standard, called by a token contract on every transfer. It never sees a user's identity — only a proof that eligibility rules are satisfied.

02

Native ZK Primitives (Protocol 25)

Stellar's "X-Ray" upgrade added BN254 pairing checks and Poseidon hashing as native host functions. Combined with SLP-4's cost cuts, per-transaction proof verification is cheap enough to run at scale for the first time.

03

Threshold-Gated Safeguards

Every powerful action — revocation, disclosure, clawback — requires a threshold of independent approvals plus a contest window, so no single party can act unilaterally against a user.

FAQs

Frequently asked

What is Reso?

Reso is an early-stage, open-source compliance layer for Stellar stablecoin and token issuers. It proves a transaction is KYC'd, sanctions-clear, and within limits using zero-knowledge proofs, without exposing user identity or balances on-chain.

How is this different from generic zkKYC tools?

Generic zkKYC tools are chain-agnostic and require users to already hold crypto/wallets. Reso is built specifically around Stellar's native primitives (Protocol 25's ZK host functions, native multisig, claimable balances, clawback) and Stellar's own SEP standards, and bridges KYC people already completed at a Stellar anchor.

Is this audited or production-ready?

No. Reso is an early-stage scaffold with working cryptographic demos for several components, but the Soroban contract has not been compiled, audited, or deployed to mainnet. See the repository status notes for exactly what's implemented versus specified.

Where can I see the code?

Everything is open on GitHub, including the compliance hook contract, ZK circuit specs, and eleven advanced feature specs (recursive proof folding, threshold oracle, timelock disclosure, revocation, FHE limits, formal verification, SEP-8, claimable balances, clawback, post-quantum readiness, and anchor KYC bridging).

This is an open, early-stage project.

There's no sales team or demo to book — just a public repository with working code, specs, and an honest account of what's built versus what's still a design doc.
